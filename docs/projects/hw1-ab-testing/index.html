A/B Testing a Call to Action

Simulating Key Ideas from Classical Frequentist Statistics

Author

Ariel Yuan

Published

April 15, 2026

Introduction

Imagine you run a website and your landing page includes a spot where visitors can sign up for your newsletter. You’ve been using a single call to action (CTA) for months, but you wonder whether different wording might drive more sign-ups. You come up with two options:

  • CTA A: “Sign up for our newsletter here!”
  • CTA B: “Stay up to date by signing up!”

Which one performs better? Rather than relying on intuition, you decide to run an A/B test — a simple randomized experiment in which each visitor to your site is randomly assigned to see one of the two CTAs. By comparing the sign-up rates between the two groups, you can make a data-driven decision about which wording is more effective.

In this post, we walk through the key ideas behind classical frequentist statistics using this A/B test as our running example. Along the way, we will simulate data, demonstrate the Law of Large Numbers and the Central Limit Theorem, use the bootstrap to estimate uncertainty, perform a hypothesis test, connect it all to linear regression, and explore the pitfalls of “peeking” at results too early.

The A/B Test as a Statistical Problem

Each visitor to our site either signs up or doesn’t, so we can model each visitor’s outcome as a draw from a Bernoulli distribution. Let \(\pi\) denote the probability of signing up. Because the CTA a visitor sees may influence this probability, we allow the parameter to differ across groups: visitors who see CTA A have sign-up probability \(\pi_A\), and those who see CTA B have sign-up probability \(\pi_B\).

The quantity we care about is the difference in sign-up rates:

\[ \theta = \pi_A - \pi_B \]

If \(\theta > 0\), CTA A is more effective; if \(\theta < 0\), CTA B wins; and if \(\theta = 0\), there is no difference. Since we cannot observe \(\pi_A\) and \(\pi_B\) directly, we estimate them from data. Our estimator of \(\theta\) is the difference in sample proportions:

\[ \hat\theta = \bar{X}_A - \bar{X}_B \]

Because each observation is either 0 or 1, the sample mean in each group is simply the proportion of visitors who signed up: \(\bar{X}_A = \hat\pi_A\) and \(\bar{X}_B = \hat\pi_B\).

Simulating Data

In the real world, we would not know the true values of \(\pi_A\) and \(\pi_B\) — the whole point of running an experiment is to learn about them. But for this exercise, we set them ourselves so we can study how our estimator behaves when we know the ground truth.

Let us suppose \(\pi_A = 0.22\) and \(\pi_B = 0.18\), giving a true difference of \(\theta = 0.04\). We simulate 1,000 visitors in each group:

pi_A = 0.22
pi_B = 0.18
theta_true = pi_A - pi_B
n = 1000

# Simulate 1,000 draws from each Bernoulli distribution
cta_a = np.random.binomial(1, pi_A, size=n)
cta_b = np.random.binomial(1, pi_B, size=n)

print(f"CTA A: {cta_a.sum()} sign-ups out of {n} visitors (sample proportion = {cta_a.mean():.3f})")
print(f"CTA B: {cta_b.sum()} sign-ups out of {n} visitors (sample proportion = {cta_b.mean():.3f})")
print(f"Estimated θ̂ = {cta_a.mean() - cta_b.mean():.4f}  (true θ = {theta_true})")
CTA A: 216 sign-ups out of 1000 visitors (sample proportion = 0.216)
CTA B: 178 sign-ups out of 1000 visitors (sample proportion = 0.178)
Estimated θ̂ = 0.0380  (true θ = 0.04000000000000001)

The Law of Large Numbers

The Law of Large Numbers (LLN) tells us that as the sample size grows, the sample mean converges to the population mean. This is the foundation of our approach: it guarantees that \(\hat\theta = \bar{X}_A - \bar{X}_B\) will get closer and closer to the true \(\theta\) as we collect more data.

To see this in action, we take the element-wise difference between each paired CTA A and CTA B draw, then compute the running average as we include more and more observations.

diffs = cta_a - cta_b
cumulative_avg = np.cumsum(diffs) / np.arange(1, n + 1)

The plot illustrates the LLN nicely. With only a handful of observations, the running average bounces around erratically — a few lucky or unlucky sign-ups can swing the estimate dramatically. But as the sample size grows, the estimate settles down and stabilizes near the true difference of 0.04. This is exactly why we need a large enough sample before drawing conclusions from an A/B test.

Bootstrap Standard Errors

Now that we trust our point estimate, the next question is: how precise is it? A single number like \(\hat\theta = 0.04\) is not very informative without some measure of uncertainty. This is where the bootstrap comes in.

The bootstrap is a resampling method that estimates the variability of a statistic without needing a closed-form formula. The idea is simple: we resample (with replacement) from our observed data many times, compute the statistic of interest on each resample, and use the standard deviation of those resampled statistics as our estimate of the standard error.

n_boot = 1000
boot_thetas = np.empty(n_boot)

for i in range(n_boot):
    boot_a = np.random.choice(cta_a, size=n, replace=True)
    boot_b = np.random.choice(cta_b, size=n, replace=True)
    boot_thetas[i] = boot_a.mean() - boot_b.mean()

boot_se = boot_thetas.std(ddof=1)

We can compare this to the analytical standard error for the difference in two independent Bernoulli proportions:

\[ SE(\hat\theta) = \sqrt{\frac{\hat\pi_A(1 - \hat\pi_A)}{n_A} + \frac{\hat\pi_B(1 - \hat\pi_B)}{n_B}} \]

pi_a_hat = cta_a.mean()
pi_b_hat = cta_b.mean()
theta_hat = pi_a_hat - pi_b_hat

analytical_se = np.sqrt(pi_a_hat * (1 - pi_a_hat) / n + pi_b_hat * (1 - pi_b_hat) / n)

print(f"Bootstrap SE:   {boot_se:.5f}")
print(f"Analytical SE:  {analytical_se:.5f}")
Bootstrap SE:   0.01766
Analytical SE:  0.01777

The two standard errors are very close, confirming that the bootstrap provides a reliable estimate of uncertainty even without relying on analytical formulas.

Using the bootstrap standard error, we construct a 95% confidence interval for \(\theta\):

ci_lower = theta_hat - 1.96 * boot_se
ci_upper = theta_hat + 1.96 * boot_se

print(f"Point estimate θ̂ = {theta_hat:.4f}")
print(f"95% CI: [{ci_lower:.4f}, {ci_upper:.4f}]")
Point estimate θ̂ = 0.0380
95% CI: [0.0034, 0.0726]

This means that, if we were to repeat this experiment many times, about 95% of the resulting intervals would contain the true value of \(\theta\). In this particular realization, the interval gives us a sense of the plausible range for the true difference in sign-up rates.

The Central Limit Theorem

The Central Limit Theorem (CLT) is one of the most powerful results in statistics. It states that regardless of the shape of the underlying population distribution, the sampling distribution of the sample mean (and therefore of our estimator \(\hat\theta\)) becomes approximately Normal as the sample size grows.

To visualize this, we repeatedly draw fresh samples of varying sizes, compute \(\hat\theta\) each time, and plot the distribution of those estimates:

sample_sizes = [25, 50, 100, 500]
n_simulations = 1000
results = {}

for sz in sample_sizes:
    theta_estimates = np.empty(n_simulations)
    for i in range(n_simulations):
        a_draws = np.random.binomial(1, pi_A, size=sz)
        b_draws = np.random.binomial(1, pi_B, size=sz)
        theta_estimates[i] = a_draws.mean() - b_draws.mean()
    results[sz] = theta_estimates

At small sample sizes (n = 25), the distribution of \(\hat\theta\) is rough, chunky, and somewhat irregular — the Bernoulli data can only produce a limited set of distinct sample proportions, so the histogram looks jagged. As n increases, the histogram becomes progressively smoother and more bell-shaped. By n = 500, the distribution is nearly indistinguishable from a Normal curve. This is the CLT at work: it doesn’t matter that our underlying data are binary (0s and 1s); with enough observations, the sampling distribution of the mean difference is well-approximated by a Gaussian.

Hypothesis Testing

Now that the CLT has shown us that \(\hat\theta\) is approximately Normal for large samples, we can perform a formal hypothesis test. We set up:

  • Null hypothesis: \(H_0: \theta = 0\) (the two CTAs produce the same sign-up rate)
  • Alternative hypothesis: \(H_1: \theta \neq 0\) (the sign-up rates differ)

The goal is to assess whether the data provide enough evidence to reject \(H_0\).

To do this, we standardize our estimate. Under the null, \(\hat\theta\) has mean 0 and standard deviation \(SE(\hat\theta)\), so:

\[ z = \frac{\hat\theta - 0}{SE(\hat\theta)} \]

Why do we standardize? Three steps of logic:

  1. By the CLT, \(\hat\theta\) is approximately Normal for large samples. Under \(H_0\), its mean is 0.
  2. Dividing a Normal random variable by its standard deviation yields a standard Normal \(N(0,1)\). So under \(H_0\), \(z \;\dot\sim\; N(0,1)\).
  3. Because we know the distribution of \(z\) under the null, we can compute a p-value: the probability of observing a test statistic as extreme or more extreme than what we actually got, assuming \(H_0\) is true.

A note on terminology: this procedure is sometimes called a “t-test,” which can be a bit misleading. The classical t-test arises when data are drawn from a Normal distribution and the variance must be estimated — in that case the ratio follows an exact \(t\)-distribution. Here, our data are Bernoulli (not Normal), so there is no exact \(t\)-distribution result. What we rely on is the CLT’s approximate Normality, making this closer to a z-test. In practice, for large samples, the \(t\)-distribution and the standard Normal are nearly identical, so the distinction has little practical consequence — but it is important to understand the conceptual difference.

z_stat = theta_hat / analytical_se
p_value = 2 * (1 - stats.norm.cdf(abs(z_stat)))

print(f"θ̂ = {theta_hat:.4f}")
print(f"SE = {analytical_se:.5f}")
print(f"z-statistic = {z_stat:.4f}")
print(f"p-value = {p_value:.4f}")
θ̂ = 0.0380
SE = 0.01777
z-statistic = 2.1388
p-value = 0.0325
At α = 0.05, we reject the null hypothesis (p = 0.0325 < 0.05).
The data provide sufficient evidence that the two CTAs have different sign-up rates.

The T-Test as a Regression

It turns out that the two-sample test we just performed is mathematically equivalent to a simple linear regression. This connection is worth understanding because it means the same machinery — and the same software — that we use for regression can also perform hypothesis tests, and it generalizes naturally to more complex experimental designs with additional controls or treatment arms.

Here is the setup: we stack the CTA A and CTA B data into a single dataset with two columns — an outcome \(Y_i\) (1 if the visitor signed up, 0 otherwise) and an indicator \(D_i\) (1 if they saw CTA A, 0 if CTA B). Then we fit:

\[ Y_i = \beta_0 + \beta_1 D_i + \varepsilon_i \]

The coefficients have clean interpretations:

  • \(\beta_0\) is the expected sign-up rate when \(D_i = 0\), i.e., the mean of the B group (\(\pi_B\)).
  • \(\beta_0 + \beta_1\) is the expected sign-up rate when \(D_i = 1\), i.e., the mean of the A group (\(\pi_A\)).
  • Therefore \(\beta_1 = \pi_A - \pi_B = \theta\), and the OLS estimate \(\hat\beta_1\) is numerically identical to \(\hat\theta\).
Y = np.concatenate([cta_a, cta_b])
D = np.concatenate([np.ones(n), np.zeros(n)])

X = sm.add_constant(D)
model = sm.OLS(Y, X).fit()
print(model.summary())

As expected, the regression coefficient \(\hat\beta_1\) matches our earlier estimate \(\hat\theta\) exactly. The standard errors and test statistics are very close, with small differences arising because OLS uses a single pooled variance estimate while our two-sample formula assumed separate variances for each group.

Why does this equivalence matter? Because the regression framework is far more flexible. If we later want to control for visitor demographics, test three CTAs instead of two, or add interaction terms, we simply expand the regression — the same approach carries through seamlessly.

The Problem with Peeking

Your boss is impatient. Rather than waiting until all 1,000 visitors have been assigned to each group, she wants to check the results after every 100 visitors per group — peeking at \(n = 100, 200, 300, \ldots, 1000\). If any of those 10 checks comes back statistically significant at \(\alpha = 0.05\), she wants to declare a winner and stop the experiment early.

This is a common temptation, but it is deeply problematic under classical frequentist statistics. Here is why: a single test at \(\alpha = 0.05\) has a 5% chance of producing a false positive (rejecting \(H_0\) when it is actually true). But each peek is another independent opportunity for a false alarm. Across 10 peeks, the probability of at least one false positive is much higher than 5%.

Let us demonstrate this with a simulation. We set up a world where the null hypothesis is true — both CTAs have the same sign-up rate (\(\pi_A = \pi_B = 0.20\)) — and see how often peeking leads us to incorrectly declare a significant result.

n_experiments = 10000
peek_points = np.arange(100, 1100, 100)  # 100, 200, ..., 1000
false_positives = 0

for _ in range(n_experiments):
    a_draws = np.random.binomial(1, 0.20, size=1000)
    b_draws = np.random.binomial(1, 0.20, size=1000)

    any_significant = False
    for peek_n in peek_points:
        a_sample = a_draws[:peek_n]
        b_sample = b_draws[:peek_n]

        p_a = a_sample.mean()
        p_b = b_sample.mean()
        diff = p_a - p_b

        se = np.sqrt(p_a * (1 - p_a) / peek_n + p_b * (1 - p_b) / peek_n)
        if se > 0:
            z = diff / se
            p_val = 2 * (1 - stats.norm.cdf(abs(z)))
            if p_val < 0.05:
                any_significant = True
                break

    if any_significant:
        false_positives += 1

peeking_fpr = false_positives / n_experiments
print(f"False positive rate with peeking (10 peeks): {peeking_fpr:.3f}")
print(f"Nominal false positive rate (single test):   0.050")
print(f"Inflation factor: {peeking_fpr / 0.05:.1f}x")
False positive rate with peeking (10 peeks): 0.199
Nominal false positive rate (single test):   0.050
Inflation factor: 4.0x

The result is striking. Instead of the nominal 5% false positive rate, peeking inflates it to roughly 15–20% — meaning that in a world where there is truly no difference between the two CTAs, we would incorrectly declare a winner about one in every five or six experiments.

This is why proper experimental design matters. If you want to peek at results during an experiment, you need to use methods specifically designed for sequential testing (such as group sequential designs or always-valid p-values) that control the overall Type I error rate across multiple looks. Simply running a standard hypothesis test at each checkpoint and stopping at the first significant result will fool you far more often than you might expect.
